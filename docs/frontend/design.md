# 前端详细设计方案

> 本文档基于《研途有我 —— 项目构想文档》，对前端 Web 应用的技术选型、页面结构、交互设计、组件规划进行详细设计。
> 
> **决策记录**：经讨论确认，前端采用 React 生态（Next.js），优先完成信息聚合模块的 MVP。

---

## 一、技术选型

### 1.1 确定技术栈

| 类别             | 选型                              | 理由                                                         |
| ---------------- | --------------------------------- | ------------------------------------------------------------ |
| 框架             | **Next.js 14+ (App Router)**      | SSR/SSG 支持利于 SEO（信息聚合页需要被搜索引擎收录）；React 生态成熟；App Router 提供更好的布局嵌套与数据获取模式 |
| UI 组件库        | **shadcn/ui + Radix UI**          | 基于 Radix 无头组件，高度可定制，视觉上限远高于 Ant Design；与 Tailwind 深度集成；社区活跃，设计感强 |
| 样式方案         | **Tailwind CSS**                  | 原子化 CSS，快速开发，与 shadcn/ui 天然配合                    |
| 动画             | **Framer Motion**                 | React 生态最强动画库，页面转场、微交互、列表动画等              |
| 状态管理         | **Zustand**                       | 轻量、简洁，适合中小型项目；无 boilerplate                    |
| 请求库           | **Axios + SWR**                   | Axios 处理 API 请求；SWR 提供缓存、重新验证、分页等能力       |
| 图表             | **ECharts**                       | 院校数据可视化（录取趋势、分数分布等）                         |
| 图标             | **Lucide React**                  | 与 shadcn/ui 配套的图标库，风格统一                            |
| 字体             | **Geist Font / 思源黑体**          | Geist 用于英文，思源黑体用于中文，现代感强                     |
| 部署             | **自建 Nginx + Docker**           | 国内用户为主，Vercel 访问不稳定；Docker 容器化便于运维          |

> 💡 **为什么选 shadcn/ui 而非 Ant Design？**
> - Ant Design 偏向后台管理系统风格，视觉上限有限，定制成本高
> - shadcn/ui 基于 Radix 无头组件，样式完全由 Tailwind 控制，可以做出更精致、更现代的 UI
> - shadcn/ui 的组件代码直接复制到项目中，完全可控，不受版本升级影响
> - 配合 Framer Motion 可以实现流畅的动画效果，整体体验更接近一线产品

## 二、MVP 优先级规划

采用分阶段交付策略，**MVP 聚焦信息聚合模块**：

### 2.1 优先级矩阵

| 优先级 | 模块                                     | 说明                                       | 预计周期   |
| ------ | ---------------------------------------- | ------------------------------------------ | ---------- |
| **P0** | 首页 + 夏令营/预推免列表与详情 + 院校库    | 核心价值，与爬虫系统直接对接                 | 第 1-4 周  |
| **P1** | 导师库 + 用户中心（收藏/登录）            | 增强用户粘性                                | 第 5-6 周  |
| **P2** | AI 简历工坊 + 择校推荐                    | 增值功能，可后期迭代                         | 第 7-10 周 |
| **P3** | 社群 + 模拟面试 + 心理支持 + 综合规划      | 复杂度高，用户量起来后再做                   | 后续迭代   |
| **P3.5** | 知识库 + 个人知识库/收藏 + 进度中心       | 内容生态与个人成长追踪                       | 后续迭代   |
| **P4** | 悬浮球 AI Bot + 多主题/动态背景 + 关于页面 | 体验增强与品牌建设                           | 后续迭代   |
| **P4.5** | 知识图谱可视化（个人知识库 + 院校关系 + 保研知识体系） | 以图谱形式展示知识关联，提升信息发现效率 | 后续迭代 |
| **P5** | 真实院校图标 + 多渠道爬虫 + UI 统一优化    | 细节打磨与数据源扩展                         | 长期迭代   |

### 3.2 页面布局

采用经典的**顶部导航 + 内容区**布局，简洁现代：

```
┌──────────────────────────────────────────────┐
│  Logo    导航菜单                    用户头像  │  ← 顶部导航栏 (Header)，毛玻璃效果
├──────────────────────────────────────────────┤
│                                              │
│              主内容区 (max-width: 1280px)      │
│              居中显示，两侧留白                │
│                                              │
├──────────────────────────────────────────────┤
│                  底部 (Footer)                │
└──────────────────────────────────────────────┘
```

**不同页面的布局变体**：
- **首页**：全宽 Hero + 居中内容区
- **信息列表页**：顶部筛选栏 + 卡片网格/列表
- **院校详情**：左侧信息卡片（sticky）+ 右侧详细内容
- **AI 辅导页面**：全屏沉浸式布局（面试房间）或左右分栏（简历编辑）
- **社群页面**：三栏布局（左侧导航 + 中间内容流 + 右侧热门/推荐）

## 四、核心页面设计（MVP 范围）

### 4.1 首页

**目标**：第一时间传达产品价值，引导用户进入核心功能。

**设计风格**：现代、简洁、信息密度适中。参考 Linear / Vercel 官网的设计语言。

| 区域             | 内容                                                       | 交互                           |
| ---------------- | ---------------------------------------------------------- | ------------------------------ |
| Hero Section     | 品牌 Slogan + 全站搜索框 + CTA 按钮                        | 搜索框支持热词提示；背景渐变动画 |
| 最新动态         | 横向滚动卡片，展示最近 7 天的夏令营/预推免通知               | 自动轮播 + 手动滑动             |
| 热门院校         | 网格卡片（3×2），展示热门 985 院校及其最新招生信息            | Hover 放大效果；点击进入详情     |
| 功能入口         | 图标 + 文字入口：夏令营、预推免、院校库、导师库               | 图标微动画                      |
| 数据统计         | 已收录 XX 所院校 / XX 条通知 / XX 位导师                    | 数字滚动动画（进入视口时触发）   |

## 五、API 接口设计（MVP 范围）

### 5.1 通知相关

```
# 通知列表（夏令营/预推免）
GET /api/notices
  Query Params:
    - type: string          # "summer_camp" | "pre_admission" | "all"
    - school_level: string  # "985" | "211" | "double_first_class"
    - province: string      # 省份名称
    - discipline: string    # 学科门类
    - status: string        # "open" | "closing_soon" | "closed"
    - keyword: string       # 搜索关键词
    - sort: string          # "latest" | "deadline" | "hot"
    - page: number          # 页码，默认 1
    - size: number          # 每页条数，默认 20
  Response: {
    total: number,
    items: NoticeItem[],
    filters: { provinces: [], disciplines: [], ... }
  }

# 通知详情
GET /api/notices/:id
  Response: NoticeDetail

# 首页最新动态
GET /api/notices/latest
  Query: { limit: number }  # 默认 10
  Response: NoticeItem[]
```

### 5.2 院校相关

```
# 院校列表
GET /api/schools
  Query Params:
    - level: string         # "985" | "211" | "double_first_class"
    - province: string
    - keyword: string
    - sort: string          # "name" | "notice_count"
    - page: number
    - size: number
  Response: { total, items: SchoolItem[] }

# 院校详情
GET /api/schools/:id
  Response: SchoolDetail (含学院列表)

# 院校下的学院列表
GET /api/schools/:id/departments
  Response: DepartmentItem[]

# 院校下的通知列表
GET /api/schools/:id/notices
  Query: { page, size }
  Response: { total, items: NoticeItem[] }
```

### 5.3 统计相关

```
# 首页统计数据
GET /api/stats/overview
  Response: {
    school_count: number,
    notice_count: number,
    department_count: number,
    tutor_count: number
  }
```

### 5.4 搜索

```
# 全站搜索
GET /api/search
  Query: { keyword: string, type?: "notice" | "school" | "tutor", page, size }
  Response: { total, items: SearchResult[] }
```

## 六、SSR/SSG 渲染策略

针对不同页面类型，采用不同的渲染策略以兼顾 SEO 和性能：

| 页面               | 渲染策略                    | 理由                                           |
| ------------------ | --------------------------- | ---------------------------------------------- |
| 首页               | **SSG + ISR (60s)**         | 内容相对稳定，ISR 定期更新最新动态               |
| 通知列表页         | **SSR**                     | 筛选条件动态变化，需要服务端渲染保证 SEO          |
| 通知详情页         | **SSG + ISR (3600s)**       | 发布后内容不变，静态生成 + 按需更新               |
| 院校列表页         | **SSG + ISR (86400s)**      | 院校数据变化极少，每天更新一次即可                |
| 院校详情页         | **SSG + ISR (3600s)**       | 学院列表相对稳定，通知部分按需更新                |
| 搜索结果页         | **CSR**                     | 纯动态内容，客户端渲染即可                       |
| 用户中心           | **CSR**                     | 私有数据，无需 SEO                               |
| 关于我们           | **SSG**                     | 纯静态内容                                      |

| 背景类型 | 说明 | 技术方案 |
|----------|------|----------|
| 渐变流动 | 柔和的渐变色缓慢流动 | CSS `@keyframes` + `background-position` 动画 |

#### 天气联动背景设计

```
获取用户位置（Geolocation API / IP 定位）
       │
       ▼
调用天气 API（和风天气 / OpenWeatherMap）
       │
       ▼
根据天气类型选择背景：
├── ☀️ 晴天 → 阳光粒子 + 暖色渐变
├── ☁️ 多云 → 云朵漂浮 + 柔和灰蓝
├── 🌧️ 雨天 → 雨滴下落动画 + 冷色调
├── ❄️ 雪天 → 雪花飘落 + 白色主题
├── 🌫️ 雾天 → 毛玻璃效果增强 + 朦胧感
└── 🌙 夜晚 → 星空背景 + 暗色主题
```

#### 技术实现要点

- 主题切换：CSS 变量 + `data-theme` 属性，Zustand 存储用户偏好
- 动态背景：独立 Canvas 层，`position: fixed; z-index: -1`，不影响页面性能
- 天气 API：和风天气免费版（每日 1000 次），缓存到 localStorage（1 小时过期）
- 性能优化：背景动画使用 `requestAnimationFrame`，低电量模式自动降级为静态背景

#### 图谱数据结构

```typescript
// 图谱节点
interface GraphNode {
  id: string;
  type: "school" | "department" | "tutor" | "question" | "experience" | "note" | "tag" | "notice";
  label: string;
  description?: string;
  metadata?: Record<string, any>;  // 额外信息（如院校层次、导师方向等）
  weight?: number;  // 节点权重（影响大小）
  x?: number;  // 固定位置（可选）
  y?: number;
}

// 图谱边（关系）
interface GraphEdge {
  source: string;  // 源节点 ID
  target: string;  // 目标节点 ID
  type: "belongs_to" | "related" | "tagged" | "references" | "similar";
  label?: string;
  weight?: number;  // 关系强度
}

// 图谱数据
interface KnowledgeGraph {
  nodes: GraphNode[];
  edges: GraphEdge[];
}
```

#### 性能优化

| 策略 | 说明 |
|------|------|
| 节点数量限制 | 单次渲染不超过 500 个节点，超出时按关联度裁剪 |
| 渐进式加载 | 初始只加载中心节点 + 一级关联，点击展开时加载更多 |
| WebWorker 布局 | 力导向布局计算放到 WebWorker 中，避免阻塞主线程 |
| Canvas 渲染 | 节点数量 > 100 时自动切换为 Canvas 渲染（替代 SVG） |
| 虚拟化 | 只渲染视口内的节点和边 |
| 缓存 | 图谱布局结果缓存到 localStorage，避免重复计算 |

### 15.1 UI 统一规范

| 项目 | 规范 |
